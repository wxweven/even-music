## 1. 版本概述

### 1.1 迭代背景

v2.0 实现了底部导航栏、「我的」页面、播放历史、歌单播放、播放模式五大核心功能。在实际使用中，发现以下体验和性能问题：

| 问题 | 用户影响 | 严重程度 |
|------|----------|----------|
| 播放详情页底部导航栏消失 | 用户无法在详情页直接切换到其他页面 | 高 |
| 播放/下载按钮各占一整行，视觉过重 | 页面布局臃肿，不符合主流音乐 App 设计 | 中 |
| 播放前后 UI 差异大（大播放按钮 vs 播放控件） | 体验不连贯 | 中 |
| 歌曲详情每次从网络加载 | 同一首歌重复进入仍需等待加载 | 高 |
| 音频无缓存，纯网络流式播放 | 断网后无法播放已听过的歌曲，重复播放浪费流量 | 高 |
| 收藏/历史点击播放后不跳转 | 用户需手动导航到详情页查看播放状态 | 中 |

### 1.2 迭代目标

- **体验一致**：播放页 UI 参考 Spotify/网易云音乐设计规范，保持简洁统一
- **性能提升**：歌曲元信息缓存 + 音频流缓存，减少网络请求，支持离线播放
- **操作流畅**：播放即跳转，减少用户手动操作步骤

### F8. 歌曲元信息缓存

**优先级**：P0 | **复杂度**：中

#### 3.8.1 需求描述

歌曲详情（`SongDetail`：歌名、歌手、封面URL、音频URL、歌词）目前每次进入详情页都从网络 HTML 解析获取。对于已经看过的歌曲，应使用缓存避免重复请求。

#### 3.8.2 缓存架构

```
┌──────────────────────────────────────┐
│            DetailViewModel           │
│                                      │
│  loadDetail(threadId)                │
│       │                              │
│       ├─ 查询缓存 ─→ 命中 ─→ 直接展示 │
│       │                              │
│       └─ 未命中 ─→ 网络请求           │
│                     │                │
│                     └─ 写入缓存       │
└──────────────────────────────────────┘
```

#### 3.8.3 双层缓存设计

| 层级 | 实现 | 容量 | 生命周期 | 查询耗时 |
|------|------|------|----------|----------|
| L1 内存缓存 | LRU `LinkedHashMap` | 50 首 | 进程存活期 | < 1ms |
| L2 磁盘缓存 | SharedPreferences + JSON | 200 首 | 永久（App卸载清除） | < 10ms |

**预热机制**：App 启动时自动将磁盘缓存中最近 20 首加载到内存缓存。

**淘汰策略**：
- 内存：LRU（最近最少使用）自动淘汰
- 磁盘：超过 200 首时删除最早缓存的记录

#### 3.8.4 缓存接入点

| 场景 | 缓存行为 |
|------|----------|
| `loadDetail(threadId)` — 首次进入详情页 | 命中→直接展示，跳过网络请求；未命中→loading + 网络请求 |
| `loadDetailForSongTransition(song)` — 切歌 | 命中→直接展示完整详情（含歌词）；未命中→占位符 + 后台请求 |
| `fetchDetail(threadId)` — 网络请求成功 | 自动写入缓存 |

#### 3.8.5 缓存数据字段

| 字段 | 类型 | 说明 |
|------|------|------|
| songName | String | 歌曲名 |
| artist | String | 歌手名 |
| audioUrl | String | 音频 URL |
| coverUrl | String | 封面 URL |
| lyrics | String? | 歌词文本 |
| realAudioUrl | String? | 解析后的真实音频 URL |

#### 3.8.6 验收标准

- [x] 首次访问歌曲从网络加载，响应后写入缓存
- [x] 再次访问同一首歌直接从缓存读取，无 loading
- [x] 切歌时缓存命中直接展示完整详情（含歌词），无占位符闪烁
- [x] App 重启后缓存仍有效
- [x] 超过容量上限自动淘汰旧数据

### F9. 音频流媒体缓存

**优先级**：P0 | **复杂度**：中

#### 3.9.1 需求描述

当前音频播放采用纯网络流式播放，存在以下问题：
- 同一首歌每次播放都重新从网络拉取
- 播放过程中断网后，缓冲区耗尽即停止
- 无法离线播放已听过的歌曲

#### 3.9.2 播放源优先级

```
播放一首歌时：

  1. 已下载文件？ ──→ 是 ──→ 直接播放本地文件（零延迟）
       │
       否
       │
  2. 流媒体缓存命中？ ──→ 是 ──→ 从磁盘缓存播放（秒播）
       │
       否
       │
  3. 网络流式播放 ──→ 边播放边缓存到磁盘
```

| 优先级 | 来源 | 延迟 | 需要网络 |
|--------|------|------|----------|
| 1 | 用户手动下载的本地文件（Music/HiFiTi/） | 零 | ❌ |
| 2 | ExoPlayer 自动缓存的音频数据 | 极低 | ❌ |
| 3 | 网络流式播放 | 取决于网速 | ✅ |

#### 3.9.3 缓存技术方案

| 组件 | 说明 |
|------|------|
| `SimpleCache` | Media3 提供的磁盘缓存，文件级管理 |
| `LeastRecentlyUsedCacheEvictor` | LRU 淘汰策略，超过上限自动删除最久未用的缓存 |
| `CacheDataSource.Factory` | 缓存感知的数据源，自动处理缓存读写 |
| `StandaloneDatabaseProvider` | 缓存索引数据库 |

**缓存配置**：

| 参数 | 值 | 说明 |
|------|-----|------|
| 最大缓存 | 2 GB | 约可缓存 400-500 首歌（按 4-5MB/首） |
| 存储位置 | `{appCacheDir}/audio_cache/` | 系统可自动清理 |
| 淘汰策略 | LRU | 空间不足时自动删除最久未播放的缓存 |
| 错误处理 | `FLAG_IGNORE_CACHE_ON_ERROR` | 缓存异常时自动回退到网络 |

#### 3.9.4 已下载文件优先逻辑

在构建 `MediaItem` 时，按以下顺序解析播放 URI：

1. 如果 URL 已经是 `content://` 或 `file://` 本地路径 → 直接使用
2. 查询 `MusicDownloader.getDownloadedUri(audioUrl)` → 有结果则用本地文件
3. 使用原始网络 URL（交给 `CacheDataSource` 处理缓存）

此逻辑对所有播放入口生效（搜索详情、收藏列表、播放历史），统一在 `MusicPlayerManager.buildMediaItem()` 中处理。

#### 3.9.5 验收标准

- [x] 播放过的歌曲音频自动缓存到磁盘
- [x] 再次播放同一首歌从缓存读取，秒加载
- [x] 已下载的歌曲优先使用本地文件播放
- [x] 断网后，已缓存/已下载的歌曲可正常播放
- [x] 缓存超过 2GB 自动淘汰最久未用的数据
- [x] 缓存异常时自动回退到网络播放

### F11. 收藏歌曲导出/导入

**优先级**：P1 | **复杂度**：中

#### 3.11.1 需求描述

用户希望能够将收藏的歌曲列表导出为文件进行备份，或在其他设备上通过导入文件恢复收藏。

#### 3.11.2 导出/导入格式

使用 JSON 文件，带版本号便于未来格式升级：

```json
{
  "version": 1,
  "exportTime": 1709520000000,
  "songs": [
    {
      "threadId": "xxx",
      "title": "歌曲名",
      "artist": "歌手",
      "coverUrl": "...",
      "audioUrl": "...",
      "addedAt": 1709520000000
    }
  ]
}
```

#### 3.11.3 文件读写方式

使用 Android SAF (Storage Access Framework)，无需申请存储权限：

| 操作 | SAF Contract | 说明 |
|------|-------------|------|
| 导出 | `ActivityResultContracts.CreateDocument` | 用户选择保存位置，默认文件名 `favorites_yyyyMMdd_HHmmss.json` |
| 导入 | `ActivityResultContracts.OpenDocument` | 用户选择 JSON 文件，支持 `application/json` 和 `*/*` |

#### 3.11.4 导入策略

导入文件解析成功后弹出对话框，让用户选择导入方式：

| 策略 | 行为 | 适用场景 |
|------|------|----------|
| 合并 | 保留现有收藏，仅添加新歌曲（按 `threadId` 去重） | 从其他设备补充收藏 |
| 覆盖 | 用导入的歌曲替换全部现有收藏 | 完整恢复备份 |

#### 3.11.5 UI 设计

在「我的」页面收藏 Tab 时，TopAppBar 右侧显示更多菜单（⋮），包含"导出收藏"和"导入收藏"：

```
┌──────────────────────────┐
│  我的                  ⋮  │  ← ⋮ 仅在收藏Tab显示
├─────────────┬────────────┤
│  收藏歌曲   │  播放历史   │
├─────────────┴──┬─────────┤
│                │ 导出收藏 │  ← DropdownMenu
│  ▶ 播放全部    │ 导入收藏 │
│          共5首 ├─────────┘
│  ...歌曲列表...           │
│  ┌────────┬─────────┐    │
│  │  首页  │   我的   │    │
│  └────────┴─────────┘    │
└──────────────────────────┘
```

导入后弹出策略选择对话框：

```
┌──────────────────┐
│    导入收藏       │
│                  │
│  检测到 8 首歌曲  │
│  请选择导入方式：  │
│                  │
│    取消  覆盖  合并│
└──────────────────┘
```

操作结果通过 Toast 提示：
- 导出成功："已导出 N 首歌曲"
- 导入成功："成功导入 N 首新歌曲"
- 失败："导出/导入失败: 错误信息"

#### 3.11.6 验收标准
